# 3.6a — Supervisor-Level Capability-Surface Teardown (Design)

- Date: 2026-06-16
- Task file: `tasks/tasks-aetherflow-delivery-architecture-alignment.md`, subtask 3.6
- Status: approved design, not yet implemented
- Branch: `chore/master-infra-alignment`

## Context

Strict audit (`docs/evidence/task-3-3-to-3-6-strict-audit.md`) failed subtask 3.6.
Task 3.6 requires that a failed plugin **unload its own capability surfaces and
direct dependents** while the shell and unrelated plugins keep running. The
current host supervisor only **degrades** direct dependents (`kDegraded`,
present-but-impaired); it never tears anything down. There is no native plugin
loader in `host/`, and `include/plugin_system.hpp` (frozen ABI) defines no
load/unload lifecycle.

Rather than block on a full plugin loader, 3.6 is split:

- **3.6a (this spec):** supervisor-level capability-surface teardown. A
  capability surface is modeled as a supervised out-of-process unit. Teardown =
  terminate the unit's process and mark its surfaces gone, cascading to direct
  dependents. Delivery claim: *"Failure isolation proven for supervised runtime
  units."*
- **3.6b (follow-on, required before first delivery):** native plugin loader +
  real DLL load/unload with full reverse-dependency-chain quiesce of workers,
  handles, and shared-memory mappings. Delivery claim: *"PRD plugin unload on
  crash."* Documented here and in the task file; **not** built now.

## Goals / Non-Goals

**Goals (3.6a):**
- A failed unit (budget-exhausted `kFailed`) and its direct dependents are torn
  down: OS process terminated, surfaces marked unavailable.
- Shell (not a supervised unit) and unrelated units are untouched.
- Frozen contracts unchanged: `proto/capture.proto`, `include/plugin_system.hpp`,
  and the `plugins::RuntimeState` enum.

**Non-Goals (deferred):**
- Real DLL load/unload / native plugin loader (3.6b).
- Transitive (multi-hop) dependency-chain teardown — 3.6a is one hop (direct
  dependents), matching task wording and the existing `direct_dependents` model.
- Auto-restoring dependents on recovery — that is 3.7.
- Exposing `torn_down` over gRPC / to the shell — that is 3.8. `torn_down` stays
  **native-only** until then; no fake `"UNLOADED"` `runtime_state` string is
  emitted anywhere.

## Constraints (non-negotiable)

- `plugins::RuntimeState` is frozen — no new enum value. Teardown is represented
  with `kFailed` plus a new supervisor-only boolean.
- `proto/capture.proto` frozen — no new RPC/message. No control-plane surface for
  `torn_down` in 3.6a.
- C++ and Python never share a process; plugins/workers are out-of-process units.
- TDD: native contract tests first, via the existing g++/clang++ direct-compile
  pattern in `tests/contracts/test_native_harness.py` (POSIX paths; short-circuit
  `return 0` under `_WIN32`).
- Hermetic tests; no FS writes outside `tmp_path`.

## State Representation

Add a supervisor-only field to `UnitSnapshot` (defined in
`include/supervisor.hpp` — NOT the frozen ABI):

```
bool torn_down;   // default false
```

| Condition                    | state         | torn_down |
| ---------------------------- | ------------- | --------- |
| Healthy                      | `kRunning`    | false     |
| Heartbeat degradation        | `kDegraded`   | false     |
| Recoverable crash            | `kRecovering` | false     |
| Budget exhausted (auto)      | `kFailed`     | **true**  |
| Explicit `TearDownUnit`      | `kFailed`     | **true**  |
| Intentional stop (`StopUnit`)| `kFailed`     | **true**  |

`kDegraded + torn_down=false` (impaired, present) stays strictly distinct from
`kFailed + torn_down=true` (gone).

## API Surface (`IWorkerSupervisor`)

New:
- `void TearDownUnit(UnitId id)` — terminate the unit's process, set
  `kFailed + torn_down`, then cascade to **direct dependents only** in
  reverse-dependency order (dependents first, then provider), terminating each
  dependent's process and setting `kFailed + torn_down`. Idempotent: re-tearing
  an already torn-down unit is a no-op. Unrelated units untouched.

Changed:
- `StopUnit(id, grace)` — **delegates to teardown semantics** (intentional stop =
  unload), not degrade-only. Sets `kFailed + torn_down` and cascades to direct
  dependents, replacing the current `DegradeDirectDependents` call.
- `RestartUnit(id)` — on successful relaunch, **clears `torn_down`** and returns
  the unit to `kRunning`. Does **not** auto-restore dependents (3.7).
- Budget path (`ApplyBudget`/`EnforceRestartBudget`) — when it drives a unit to
  `kFailed`, it triggers the teardown cascade automatically. `kRecovering`
  outcomes leave `torn_down=false`.

Unchanged:
- `DegradeDirectDependents` — retained for **heartbeat degradation only**
  (`kDegraded`, `torn_down=false`). Teardown is a separate code path; the two do
  not share an implementation.

## Trigger Matrix

| Trigger                              | Teardown? | torn_down |
| ------------------------------------ | --------- | --------- |
| Budget exhausted → `kFailed` (auto)  | Yes       | true      |
| Explicit `TearDownUnit` (host/loader)| Yes       | true      |
| `StopUnit` (intentional)             | Yes       | true      |
| Recoverable crash → `kRecovering`    | No        | false     |
| Missed heartbeats → `kDegraded`      | No        | false     |

Until 3.6b exists, crash-driven unload is reached **only** via the budget→`kFailed`
auto path or an explicit `TearDownUnit` call from the host/loader. There is no
plugin-loader callback wiring real DLL crashes to teardown in 3.6a.

## Teardown Algorithm

1. If target already `torn_down`, no-op.
2. Collect target's direct dependents (existing `direct_dependents`).
3. Reverse-dependency order: process dependents first (workers/handles quiesce
   before their provider), then the target.
4. For each: `process->Terminate(grace)`, `state = kFailed`, `torn_down = true`.
5. Unrelated units (not in `direct_dependents`) are never visited.

## Testing (native contract, g++/clang++)

New tests in `tests/contracts/test_native_harness.py` (inline `.cpp`, mock
`IProcessHandle` via `InjectProcess` recording `Terminate` calls):

1. Budget exhaustion auto-tears-down the failed unit + its direct dependents;
   both end `kFailed + torn_down=true`; an unrelated unit stays
   `kRunning + torn_down=false`.
2. Explicit `TearDownUnit` terminates the unit's and dependents' processes
   (mock records `Terminate`).
3. Reverse-dependency order asserted (dependent torn down before provider).
4. Recoverable crash (`kRecovering`) leaves `torn_down=false`.
5. `StopUnit` produces `kFailed + torn_down=true` and tears down direct
   dependents.
6. `RestartUnit` success clears `torn_down`, returns `kRunning`, and does NOT
   restore dependents.
